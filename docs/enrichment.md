# Enrichment — Tag Taxonomy & Classification

## What It Does

`openaugi enrich` builds a clean faceted tag taxonomy from your messy source tags and classifies documents against it. Source tags are never modified — computed tags live in a separate metadata layer.

## How It Works

### Two layers of tags

| | Source Tags | Computed Tags |
|---|---|---|
| **Origin** | Extracted from markdown (`#tag` in content) | Generated by agent or rules |
| **Stored as** | `block.tags` | `block.metadata["computed_tags"]` |
| **Mutability** | Never modified — source truth | Regenerated on enrich |
| **Accessed via** | `block.tags` | `block.effective_tags` (prefers computed, falls back to source) |

### Faceted taxonomy

Tags are organized into facets:

- **area/** — domain of life (health, family, work, openaugi, content, self)
- **type/** — what kind of artifact (idea, task, insight, memory, journal, reference)
- **status/** — lifecycle, ONLY for tasks/workstreams (active, done, parked, review)
- **topic/** — subject matter, open-ended (discovered from data)

### The flow

```
openaugi enrich --tags
  │
  ├─ 1. Export: tag inventory + doc summaries → ~/.openaugi/enrich/
  ├─ 2. Agent: Claude reads inventory, proposes taxonomy, writes rules
  │     → ~/.openaugi/enrich/tag_rules.json
  │     → ~/.openaugi/enrich/doc_tags.json (optional)
  └─ 3. Apply: openaugi enrich --apply
        → reads rules, normalizes source tags → computed_tags on blocks
        → reads doc classifications → trickle tags to entries
```

## Usage

### Default: agent path (recommended)

```bash
# Step 1: Export inventory and launch Claude agent
openaugi enrich --tags

# Step 2: Agent analyzes tags, writes rules (runs automatically if claude CLI found)

# Step 3: Apply results
openaugi enrich --apply
```

### Fallback: API calls (headless/automated)

```bash
# All-in-one: discover + apply + classify via API
openaugi enrich --tags --model gpt-5.4-nano
```

### Just apply existing rules

```bash
# Re-apply rules after re-ingest (no LLM needed)
openaugi enrich --apply
```

## Files

| File | Purpose |
|---|---|
| `~/.openaugi/enrich/tag_inventory.md` | All tags with usage counts (input for agent) |
| `~/.openaugi/enrich/doc_inventory.md` | All documents with titles, folders, tags (input) |
| `~/.openaugi/enrich/tag_rules.json` | Taxonomy + merge rules + ignore patterns (output) |
| `~/.openaugi/enrich/doc_tags.json` | Document classifications (output, optional) |
| `~/.openaugi/tag_rules.json` | Canonical rules location (copied on apply) |

## Tag Rules Format

```json
{
  "taxonomy": {
    "area": ["health", "work", "openaugi", "content", "self"],
    "type": ["idea", "task", "insight", "journal", "reference"],
    "status": ["active", "done", "parked", "review"],
    "topic": ["bits-of-chris", "augmented-engineer", "jung"]
  },
  "ignore_patterns": ["^\\d+$", "^\\d{4}-\\d{2}"],
  "merge": {
    "idea/content/bits-of-chris": ["area/content", "type/idea", "topic/bits-of-chris"],
    "note-type/daily-journal": ["type/journal"],
    "active": ["status/active"]
  },
  "discovered_topics": ["openaugi", "comedy", "jung"]
}
```

## How Compile Uses Enriched Tags

After enrichment, `openaugi compile` produces cleaner output:

- Hub summaries and concept pages reflect the real taxonomy
- MCP tools (`search`, `get_context`, `recent`) filter by `effective_tags`
- The `get_index` tool shows the navigational map with clean tag names

Run `openaugi compile` after `openaugi enrich --apply` to rebuild context blocks.

## Architecture

### Modules

- `pipeline/enrich.py` — Export inventory, launch agent, apply results
- `pipeline/taxonomy.py` — Tag rules engine (load, save, apply, normalize)
- `pipeline/tag_inference.py` — Batched document classification via LLM (API path)
- `models/llms/openai.py` — OpenAI-compatible LLM provider

### Key design decisions

- **Source tags are sacred.** `block.tags` is never modified by enrichment.
- **`block.effective_tags`** transparently picks computed_tags when available, falls back to source.
- **Agent-first.** Default path uses Claude Code — no API key needed. API is a fallback.
- **Rules are mechanical.** Once the agent writes `tag_rules.json`, applying them is pure regex + dict lookup. No LLM in the apply step.
- **Incremental.** Re-running `enrich --apply` after re-ingest is cheap and fast.
